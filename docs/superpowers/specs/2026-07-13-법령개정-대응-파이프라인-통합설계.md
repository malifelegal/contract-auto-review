# 법령 개정 대응 파이프라인 — 통합 설계

- 작성일: 2026-07-13 · 상태: 초안(팀 합의 대기)
- 배경: 준법감시본부 AX 과제 보고서가 팀장 「법령-사규 확정쌍」 매칭 도구 위에 ① 개정 대응 워크리스트 ② 매칭 시각화를 얹었다. 같은 시기 팀 공통 공간에는 두 도구가 이미 존재한다 — `legal-`(법령 공포·개정 자동 감지, 정적 사이트)와 `contract-auto-review`(오프라인 계약서 리뷰, 다신호 매칭엔진 + 조문 근거 파이프라인). 세 조각이 **같은 아키텍처 DNA**(오프라인·파일교환·근거중심)를 공유하므로, 코드 병합 없이 하나의 반복 프로세스로 묶는다.
- 목표: "법령이 바뀌었다"를 감지에서 끝내지 않고 "누가·언제·무슨 근거로 어느 사규를 고쳤다"까지 잇는다. 효용성(갭·우선순위·근거)과 편의성(자동감지·승인제·설명가능·오프라인)을 동시에 올린다.

## 설계 원칙 (제약을 기능으로)

1. **오프라인·정적 유지.** 서버·AI·외부망 연결 0. 망분리 원칙에 부합하고 설치·학습비용이 없다. 세 도구 모두 이미 이 방식이다.
2. **도구는 파일로 결합.** 코드를 합치지 않는다. 각 도구는 CSV/JSON 규격(§4)만 주고받는다. 팀장 도구는 수정 대상이 아니다.
3. **전량 자동 아님, 승인제.** 기계는 *후보와 근거*를 낸다. 사람은 *승인/기각/판정*만 한다. 모든 항목은 "왜 떴는지"(매칭도 + 근거 조문)가 항상 설명된다.
4. **근거 중심.** 매칭은 매칭도(%)로, 반영 판단은 조문 원문(quote)으로 뒷받침한다. 감으로 반영하지 않는다.

## 아키텍처 — 세 레이어, 파일로 연결

```
[감지 레이어]                 [매칭 레이어]                  [처리 레이어]
legal-                        contract-auto-review           AX 워크리스트/시각화
법령 공포·개정 수집            sim.js 다신호 매칭엔진          개정↔확정쌍 join
(law.go.kr OpenAPI,           knowledge/*.yaml 근거          상태·담당·기한
 로컬 수집→JSON 커밋)          enrich.py 조문 quote 대조       CSV 내보내기
      │                             │                              ▲
      │  개정목록.csv                │  확정쌍.csv                   │
      └──────────────┬──────────────┘                              │
                     └───────  '법령[+조항]' 키로 자동 join  ───────┘
```

- **감지(legal-)**: 이미 CORS·등록IP 제약을 로컬 수집으로 우회한다. `docs/data/laws.json`이 법령 단위 공포·개정 데이터를 담는다. → `개정목록.csv` 어댑터 추가.
- **매칭(contract-auto-review)**: `sim.js`(char_wb TF-IDF·Jaccard)+`matcher.js`(tier)가 "조항↔체크" 매칭도를 이미 계산한다. 대상을 "법령조항↔사규조항"으로 바꿔 **확정쌍 초안**을 생성한다. `knowledge` 스키마·`enrich.py`가 조문 quote 대조·검수 승급을 제공한다.
- **처리(AX 워크리스트)**: 개정목록과 확정쌍을 `법령[+조항]` 키로 join → 영향 사규 목록에 상태·담당·기한을 얹는다. localStorage 저장 + CSV 내보내기로 팀 공유.

## 데이터 규격 — 파일 교환 계약 (이 문서의 핵심)

도구를 분리하는 유일한 접합면. 컬럼명은 AX 워크리스트 `normPairs`/`normAmend`가 이미 인식하는 한국어 헤더에 맞춘다(별칭 허용).

### 4.1 개정목록.csv — 감지 레이어 산출 (legal- → 처리)
| 컬럼 | 필수 | 출처(legal- 필드) | 비고 |
|---|---|---|---|
| 법령 | ✅ | `name`(법령명한글) | join 키 1 |
| 법령조항 | ⚠️ | — | 4단계까지 공란(법령 단위 join) |
| 구분 | ✅ | `revisionType`(제개정구분명) | 개정/제정/신설 |
| 개정일 | ✅ | `promulgationDate`(공포일) | |
| 시행일 | 선택 | `enforcementDate` | 우선순위 임박도 산정용 |
| 링크 | 선택 | `links[].url` | 신구조문대비표 딥링크 |

### 4.2 확정쌍.csv — 매칭 레이어 산출 (팀장 도구 또는 엔진 → 처리)
AX 문서의 4필수+2선택 규격을 그대로 채택.
| 컬럼 | 필수 | 비고 |
|---|---|---|
| 법령 / 법령조항 | ✅ | join 키 |
| 사규 / 사규조항 | ✅ | 대응 대상 |
| 소관부서 | 선택 | 처리 배정 힌트 |
| 매칭도 | 선택 | 엔진 산출 시 sim %, 수동 시 공란 |

> 민감 조문 본문은 교환하지 않는다(AX 문서 원칙 유지). quote 대조는 각 도구 내부에서만.

### 4.3 워크리스트.csv — 처리 레이어 산출 (감사·공유용)
key=`법령|법령조항|사규|사규조항` + 상태(미검토/검토중/반영완료/해당없음)·담당·기한·메모. AX `exportCsv`가 이미 생성.

## 역할 분담 (6단계 반복 루프)

| 단계 | 담당 | 산출물 |
|---|---|---|
| 1. 감지 | 🤖 legal- | 신규 개정 → 개정목록.csv |
| 2. 영향 join | 🤖 워크리스트 | 개정 ↔ 확정쌍 → 영향 사규 목록 |
| 3. **갭 판정** | 🤖 | 대응 사규 없는 개정 = "보완 필요" 최우선 플래그 |
| 4. 우선순위 | 🤖 | severity × 시행 임박 × 영향 수 → 처리순위 |
| 5. 배정·처리 | 👤 | 상태·담당·기한 입력 |
| 6. 검증·이력 | 👤 | 근거 대조 후 승급 → 감사 이력 보존 |

기계와 사람의 경계가 3·4단계에서 뚜렷하다: 기계는 갭과 순위를 *제시*, 사람은 처리·판정.

## 효용성 장치

1. **갭을 1급 산출물로.** 개정이 떴는데 대응 확정쌍이 없으면 최우선 빨간 신호(AX 매트릭스의 빈칸 + `contract-auto-review`의 `absence_check` 개념). 모니터링이 행동으로 전환되는 지점.
2. **우선순위 자동 산정.** `validate.py`의 `derive_severity(norm_type, basis)`를 재활용 — 강행규정 개정 + 시행일 임박 + 영향 사규 많음 = 최상단. 100건이 떠도 손댈 순서가 정해진다.
3. **근거 자동 첨부.** 개정 조항 옆에 연결 사규조항의 원문 quote를 붙인다(`enrich.py`의 quote 대조 기계 그대로).
4. **감사 이력.** 처리 완료 건도 "언제·누가·무슨 근거로 반영/해당없음"으로 보존 → 감독·감사 대응 자료가 부산물로 축적.

## 사용자 편의성 장치

1. **수동 월1회 → 자동 감지 + 배지.** legal- 수집분에서 신규 N건을 워크리스트가 알림(AX 문서가 인정한 약한 고리 제거).
2. **확정쌍 전량 작성 → 승인제.** 엔진이 후보쌍+매칭도 제시, 사람은 ✔/✘. 노동이 검수로 전환.
3. **항상 설명가능.** 항목마다 "매칭도 92% · 근거 제N조". 블랙박스 없음.
4. **원클릭·오프라인.** 단일 HTML, 브라우저만. 파일 열기/CSV로 팀 공유.
5. **분산 검수 내장.** `verification.json` → `merge_verifications.py` → `apply_verification.py` 흐름이 "여럿이 나눠 검수"를 이미 구현.

## 단계적 도입 로드맵

- **1단계 — 감지→워크리스트 연결.** legal-에 `scripts/export-amendments.mjs`(laws.json → 개정목록.csv). **법령 단위** join으로 시작. 리포 안에서 완결, 팀장 도구 무수정. *가장 빨리 효용 확인.*
- **2단계 — 확정쌍 반자동 생성.** sim.js 매칭엔진을 법령↔사규 축으로 돌려 확정쌍 초안(승인제).
- **3단계 — 갭·근거 대시보드.** 커버리지 갭·우선순위·근거 조문을 한 화면에(AX 매트릭스/연결맵 시각화를 실데이터로).
- **4단계 — 조항 단위 join.** 신구조문대비표 파싱으로 "어느 조가 바뀌었나"까지. 난이도 최상, 마지막.

## 갭·리스크

- **조항 granularity.** AX join 키는 (법령+조항)이나 legal-는 법령 단위까지만 안다. 조항 단위는 신구조문대비표(첨부) 파싱이 필요 → 4단계로 격리. 1~3단계는 법령 단위로 충분히 작동한다.
- **laws.json 실데이터 0건.** 현재 OC 미수집 상태. 어댑터 검증에는 등록IP에서 `scripts/fetch.mjs` 1회 실행 필요.
- **확정쌍 출처 이원화.** 팀장 수동본과 엔진 초안이 공존할 수 있다. 규격(§4.2)이 동일하므로 워크리스트는 출처를 구분하지 않고 먹되, 검수 배지로 신뢰도를 구분한다.

## 열린 결정 (팀 합의 필요)

1. 확정쌍의 1차 출처를 팀장 수동본으로 둘지, 2단계 엔진 초안으로 조기 전환할지.
2. 공유 방식: 초기 공유 엑셀/CSV → 이후 사내 시스템 배포 시점.
3. 외부 Git 공동개발 가부(보안팀 확인 — 사규 데이터 분리 전제).
