# 검토의견 지식 루프 설계안 (#4)

> 사용자 요구: 조항별 보기에서 남긴 검토의견(이상없음/검토의견/해당없음 + 코멘트)을 쌓아 **루프 형태로 지식이 정교화**되게. 폐쇄망 내부 루프 우선, 외부 LLM은 비선호 옵션. 검토자별 편차를 억지 통일하지 말고 **분포로 집계**.

## 결론 요약

**외부 LLM 없이, 폐쇄망 내부 루프만으로 실용적 성과가 난다.** verdict 데이터는 cpId(체크항목) 단위로 쌓이므로, "같은 체크항목에 여러 계약서·검토자가 어떤 판정·코멘트를 남겼나"를 집계하면 세 가지 정교화가 가능하다:
1. **코멘트 추천** — 이 항목에 자주 다는 의견 제시
2. **판정 분포** — 이 항목의 과거 판정 비율(이상없음 X% / 해당없음 Y% …)
3. **골드셋·매칭 보정** — 반복 판정으로 신뢰도·조건부화 신호 축적

편차는 통일하지 않고 그대로 분포로 보여준다(다양성 = 정보).

---

## 1. 데이터 모델 — "검토의견 코퍼스"

현재: verdict는 계약서 건별(`cr-verdict-<hash>`)로만 저장. 건 사이 축적이 없음.

신규: **검토의견 코퍼스**(cpId 단위 누적). 앱과 별도의 축적 저장소.

```
knowledgeLoop = {
  meta: { updated, contract_count, reviewer_count },
  byCheck: {
    "<cpId>": {
      counts: { 이상없음: n, 검토의견: n, 해당없음: n },   // 판정 분포
      comments: [                                          // 코멘트 이력
        { text, verdict, reviewer, date, contract_hash, count }  // 동일 텍스트는 count 병합
      ],
      lastSeen: date
    }
  }
}
```

- **cpId 단위 집계**가 핵심 — 체크항목은 유형·계약을 관통하는 안정 키.
- 코멘트는 원문 보존(정규화 안 함). 동일 텍스트만 count로 병합.
- reviewer는 파일명/검수자명(협업 merge_verifications 패턴 재사용).

## 2. 루프 사이클 (폐쇄망 내부)

```
① 검토 (조항별 보기): 판정 + 코멘트 입력 → 계약서 건별 verdict
② 내보내기: 기존 exportVerdicts(건별 JSON)
③ 코퍼스 적재: 여러 건 verdict JSON을 knowledgeLoop에 병합
   — 앱 내 "지식 적재" 또는 별도 병합(팀 협업 시 merge_verifications 확장)
④ 활용 (다음 검토): 조항별 보기·리포트에서 해당 cpId의 분포·추천코멘트 표시
   → 검토자가 참고해 더 빠르고 일관되게 판정
⑤ 축적 반복 → 분포가 안정될수록 신뢰도↑
```

핵심: ③의 적재가 ④의 활용으로 이어지는 **닫힌 루프**. LLM 개입 없음.

## 3. 세 가지 정교화 (내부 루프)

### 3-1. 코멘트 추천
- 조항별 보기에서 항목을 볼 때, 그 cpId에 과거 달린 코멘트를 count 내림차순으로 3건 제시.
- "이 항목에 자주 남긴 의견: [손해배상 상한 확인 필요 ×5] [해당없음-개인정보 미처리 ×3]"
- 클릭하면 코멘트 입력창에 채움(재사용). 검토자는 수정 가능.
- **편차 그대로 노출** — 서로 다른 의견이 여럿이면 다 보여줌.

### 3-2. 판정 분포
- 각 항목에 과거 판정 비율 배지: "과거: 이상없음 72% · 해당없음 20% · 검토의견 8% (n=25)".
- 검토 전 기대치 제공. n이 작으면(<5) "표본 적음" 표기.
- 특정 판정이 압도적이면(예: 해당없음 90%+) "이 유형 계약엔 대체로 해당없음" 힌트.

### 3-3. 골드셋·매칭 보정
- **골드 신뢰도**: 어떤 매칭(cpId↔조항 성격)이 반복적으로 "이상없음"이면 그 매칭 신뢰도↑ → 향후 자동 addressed 강화(현 tier 게이트의 골드셋 후속과제와 연결).
- **조건부화 신호**: 특정 cpId가 반복적으로 "해당없음"이면 그 항목이 해당 유형에 과잉일 수 있음 → 모듈 조건 재검토 후보로 리포트(지식 큐레이터에게).
- 자동 반영 아님 — **신호를 큐레이터에게 제시**, 사람이 지식 수정 판단(법무 하네스: 자동 지식변경 금지).

## 4. 검토자 편차 처리

- **통일 안 함.** 코멘트는 원문 보존, 판정은 분포로.
- 코멘트 추천·분포에 **검토자 수**를 함께 표시("3명 중 2명 이상없음").
- 협업(merge_verifications)의 보수적 규칙과 정합: 판정 충돌은 분포에 그대로, 승급은 별도.
- 향후(선택): 검토자 신뢰가중(경력자 의견 가중)은 과설계 위험 — 기본은 균등.

## 5. 외부 LLM 옵션 (비선호, 조건부)

내부 루프로 안 되는 것만:
- **코멘트 정규화**: 검토자별 표현을 표준 문구로 통일(예: "상한 확인"·"배상상한 봐야" → "책임상한 검토 필요").
- **신규 논점 추출**: 코멘트에서 기존 체크항목에 없는 새 논점 발견 → 신규 check 후보.

방식(동선 최소화):
- 앱에서 "코퍼스 내보내기"(JSON) → 사외에서 LLM 처리 → 결과를 지식 큐레이션에 반영.
- **앱 자체는 LLM 미연동**(오프라인 원칙 유지). 큐레이션 단계의 보조 도구로만.
- 이 경로는 필요 시에만. 기본 루프는 3장으로 완결.

## 6. 구현 순서 (제안)

1. **코퍼스 데이터 모델 + 순수 로직**(loop.js): 병합(mergeIntoCorpus)·집계(checkStats)·추천(topComments). TDD.
2. **적재 UI**: verdict JSON 여러 개 → 코퍼스 병합. 앱 내 또는 build 스크립트.
3. **활용 UI**: 조항별 보기 항목에 판정분포 배지 + 코멘트 추천(클릭 재사용).
4. **골드/조건부화 신호 리포트**: 큐레이터용 요약(자동반영 아님).
5. (선택) 코퍼스 내보내기 → 외부 LLM 경로.

## 7. 법무 하네스 준수

- 지식(체크항목·근거) 자동 변경 금지 — 루프는 **참고·신호**만, 확정 수정은 사람.
- 코멘트는 검토자 저작 — 원문 보존, AI 대필 금지.
- 코퍼스는 검토의견(사람 판단)의 집계이지 법령 해석 생성이 아님.

## 8. 폐쇄망 실현가능성 결론

| 정교화 | 내부 루프(LLM無) | 근거 |
|---|---|---|
| 코멘트 추천 | ✅ | cpId별 코멘트 count 집계 |
| 판정 분포 | ✅ | cpId별 verdict count |
| 골드셋 신뢰도 | ✅ | 반복 이상없음 매칭 가중 |
| 조건부화 신호 | ✅ | 반복 해당없음 항목 플래그 |
| 코멘트 정규화 | ❌ LLM | 표현 통일은 생성 필요 |
| 신규 논점 | ❌ LLM | 코멘트→신규 check 추출 |

→ **핵심 4개가 내부 루프로 가능.** 폐쇄망에서 성과 나옴. LLM은 고차 2개에만, 그것도 큐레이션 보조로 선택적.
