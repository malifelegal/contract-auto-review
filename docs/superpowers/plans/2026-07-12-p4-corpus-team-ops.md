# P4 코퍼스 팀 운영 — 판정파일 교환 + 취합 절차

> 지식 루프(코퍼스)가 개인 localStorage에만 쌓여 정규화 신호(curationSignals 표본≥5)에 도달하기 어려웠음. 팀원들의 검토 판정을 모으는 교환 구조와 운영 절차를 확립.

## 설계 결정: 교환 단위는 판정파일, 코퍼스는 로컬 집계 뷰
- **코퍼스끼리 병합 금지** — byCheck 카운트에는 계약별 출처가 없어, 두 코퍼스가 같은 계약을 이미 반영했으면 이중 카운트됨(멱등 불가).
- **판정파일(verdict JSON)은 contract_hash로 멱등 병합**(`mergeIntoCorpus` 기존 구현) — 같은 파일을 몇 번 반영해도 안전. 따라서 판정파일이 팀의 진실의 원천, 코퍼스는 각자 브라우저의 materialized view.

## 구현 (앱)
- 검토자 이름: 리포트 하단 입력(localStorage `cr-reviewer`) → 내보내기 meta.reviewer → 코멘트 귀속(추천 코멘트의 reviewers 표시).
- 검토의견 내보내기 파일명 유니크화: `verdicts_<유형>_<계약해시>_<검토자>_<날짜>.json` — 공유폴더 충돌 방지.
- **판정파일 일괄 반영**: 파일 다중선택 → 각각 `mergeIntoCorpus`(멱등) → 결과 메시지(병합 N건·실패 N건·코퍼스 총 N건).
- **코퍼스 백업/복원**: 브라우저 이동·유실 대비(형태 검증 후 통째 교체). 교환용 아님을 UI 문구로 명시.

## 운영 절차 (주간, 금요일 주간배치와 함께)
1. **검토자(상시)**: 계약 검토 → 조항별 판정·코멘트 → 리포트에서 ①"이 검토를 지식에 반영"(자기 코퍼스) ②"검토의견 내보내기" → 파일을 공유폴더 `verdicts/`에 저장.
2. **각자(주 1회)**: 공유폴더 `verdicts/` 전체 파일을 "판정파일 일괄 반영" — 멱등이라 전체 재반영해도 안전. 각자의 코퍼스가 팀 전체 판정으로 수렴.
3. **큐레이터(주 1회)**: 리포트 하단 "지식 정규화 후보" 패널 확인 → 조건부 강등 후보(표본≥5·해당없음 80%+)를 검토 → 타당하면 knowledge yaml에 `tier: conditional`(또는 absence_precondition) 반영 → 골드셋·빌드 → 새 dist 배포.
   - 판정 기준(P2에서 확립): 문언에 제도 채택이 드러나면 precondition, 문언 외 사실(규모·자격)로 적용 결정되면 conditional.
4. **배포**: dist/contract-review.html 갱신본을 공유 — 코퍼스는 localStorage라 앱 파일 교체와 무관하게 유지됨.
